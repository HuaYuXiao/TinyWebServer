# Redis 缓存设计文档

## 一、业务场景分析

本系统是典型的**读密集型**场景：

| 特征 | 描述 |
|:---|:---|
| 读写比 | 极高（>1000:1），成绩发布后极少量写入，海量查询 |
| 热点数据 | 热门考生的成绩可能被高频查询，存在热点 Key |
| 流量峰值 | 成绩发布瞬间流量洪峰，QPS 可能是平时的数百倍 |
| 数据规模 | 考生总量百万级，单个查询结果约 200–500 字节 JSON |
| 恶意流量 | 可能存在大量无效身份证号遍历（穿透攻击） |

## 二、核心设计原则

1. **Cache Aside 模式** — 读写分离，先写 DB 再删缓存
2. **三级防护体系** — 穿透（布隆过滤器）→ 击穿（互斥锁）→ 雪崩（随机 TTL）
3. **优雅降级** — Redis 不可用时自动回退到 MySQL 直连，不中断服务
4. **与现有架构兼容** — 沿用 RAII + `std::counting_semaphore` + 单例的连接池模式

## 三、缓存策略

### 3.1 缓存穿透防护

**问题**：恶意查询不存在的考生 ID，缓存永远不命中，每次穿透到 DB。

**方案**：布隆过滤器 + 空值缓存双层防护。

```
请求 → 布隆过滤器
         ├─ 一定不存在 → 直接返回 404（不访问 Redis/DB）
         └─ 可能存在   → 查 Redis
                           ├─ 命中 → 返回
                           ├─ 命中 __NULL__ 标记 → 返回 404
                           └─ 未命中 → 查 DB → 写入缓存
```

- **布隆过滤器**：100 万元素，1% 误判率 → 位数组约 1.2MB
- **空值缓存**：DB 返回空时写入 `__NULL__` 标记，TTL=60s
- 布隆过滤器在系统启动时从 MySQL 预热加载所有已知考生 key

### 3.2 缓存击穿防护

**问题**：热点考生成绩缓存刚过期，同时涌入大量查询，全部穿透到 DB。

**方案**：分布式互斥锁（SETNX + TTL）。

```
缓存未命中 → SETNX lock:exam:score:{key} 1 EX 10
               ├─ 获取锁 → Double-check 缓存 → 查 DB → 写缓存 → 释放锁
               └─ 未获取 → sleep(100ms) → 重试（最多5次）→ 最终降级直查 DB
```

关键设计：
- 锁 TTL=10s，防止死锁
- Double-check：获取锁后再次查缓存，避免重复重建
- 重试机制：未获取锁的线程等待并重试读缓存，而非直接穿透

### 3.3 缓存雪崩防护

**问题**：大量缓存在同一时间段集中过期，瞬时压力压垮 DB。

**方案**：随机 TTL 抖动 ±10%。

```cpp
int random_ttl(int base_ttl) {
    int jitter = base_ttl * 0.1;
    return max(10, base_ttl + uniform_dist(-jitter, jitter));
}
```

配合熔断器（Redis 不可用时快速失败）和多级 TTL（不同业务使用不同基础 TTL）。

## 四、数据结构选型

| Key 模板 | 类型 | 说明 |
|:---|:---|:---|
| `exam:score:{name}:{id_card}` | String (JSON) | 单个考生成绩缓存 |
| `lock:exam:score:{name}:{id_card}` | String (临时) | 击穿互斥锁，SETNX 自动过期 |
| 布隆过滤器 | 内存 Bitmap | 本地内存，不存 Redis |

**选型理由**：

- **选 String 而非 Hash**：成绩是一次性完整返回的 JSON，原子读写比字段级更新更常见。GET/SET 时间复杂度 O(1)。若需字段级更新 Hash 更合适，但本场景不需要。
- **布隆过滤器不存 Redis**：百万级布隆过滤器仅 ~1.2MB，本地内存操作无网络往返（<1μs vs >0.1ms）。布隆过滤器只增不删，无需 Redis 持久化。

## 五、缓存更新策略（Cache Aside）

```
读:  GET redis → 命中返回 / 未命中 → SELECT mysql → SETEX redis → 返回
写:  UPDATE mysql → DELETE redis（下次读时重建）
```

**为什么选 Cache Aside？**

对于考研成绩系统，成绩是**一次性写入、长期读取**的数据。Cache Aside 模式下写入只需删除缓存（或直接不删除，因为成绩不变），读取按需重建。

| 考量 | Cache Aside | Read/Write Through |
|:---|:---|:---|
| 缓存与 DB 一致性 | 最终一致 | 强一致（同步写） |
| 写性能 | 高（仅删缓存） | 低（同步写缓存+DB） |
| 架构复杂度 | 低 | 需缓存代理层 |
| 适用场景 | 读多写少（本系统） | 读写均衡 |

## 六、内存管理与过期策略

| 配置项 | 值 | 理由 |
|:---|:---|:---|
| Key 命名 | `exam:score:{name}:{id_card}` | 业务前缀隔离，冒号分段 |
| 基础 TTL | 3600s（1 小时） | 成绩不会变，够长且允许内存回收 |
| 空值 TTL | 60s | 防穿透标记，短 TTL 减少内存占用 |
| maxmemory-policy | `allkeys-lru` | 内存满时淘汰最少使用的 key |
| maxmemory | 物理内存的 50% | 留足空间给 OS 和布隆过滤器 |

## 七、容错降级方案

```
┌─────────────────────────────────────────────────┐
│  Redis 正常                                      │
│  请求 → Redis(命中) → 返回                        │
│  请求 → Redis(未命中) → Lock → DB → 返回          │
├─────────────────────────────────────────────────┤
│  Redis 不可用                                    │
│  连续失败5次 → 熔断器 OPEN                        │
│  OPEN：快速失败，不走 Redis                        │
│  30s 后 → HALF_OPEN → 放行3个探测请求              │
│  探测成功 → CLOSED   探测失败 → OPEN              │
├─────────────────────────────────────────────────┤
│  最坏情况                                        │
│  熔断器 OPEN + 获取不到 Redis 连接                 │
│  → 直接查询 MySQL 返回（有损服务，不中断）          │
└─────────────────────────────────────────────────┘
```

## 八、监控告警方案

| 指标 | 告警阈值 | 说明 |
|:---|:---|:---|
| Redis 命中率 | < 90% | 缓存效率不足 |
| 布隆过滤器拦截率 | 突然飙升 > 50% | 可能遭受穿透攻击 |
| 熔断器 OPEN 次数 | > 0 | Redis 服务异常 |
| DB 直连查询 QPS | > 基准值的 2 倍 | 缓存大面积失效 |
| 缓存重建耗时 P99 | > 100ms | DB 慢查询 |
| Redis 连接池空闲率 | < 20% | 连接池不足 |

## 代码实现

### 文件清单

| 文件 | 功能 |
|:---|:---|
| `redis_connection_pool.h/.cpp` | Redis 连接池（RAII + semaphore） |
| `bloom_filter.h` | 布隆过滤器（防穿透） |
| `circuit_breaker.h` | 熔断器（CLOSED / OPEN / HALF_OPEN） |
| `redis_cache.h/.cpp` | 缓存工具类（三级防护集成） |
| `exam_score_handler.h` | 业务层集成示例 |

### 关键模块说明

**连接池** (`redis_connection_pool.h`) — 完全对照 MySQL 连接池的 RAII 模式，使用 `std::counting_semaphore` 控制并发，带 60s 冷却健康检查和自动重连。

**布隆过滤器** (`bloom_filter.h`) — 双哈希函数 `g_i(x) = (h1(x) + i·h2(x)) % m`，FNV-1a + std::hash。根据预期元素数和误判率自动计算最优位数组大小和哈希次数。

**缓存读取核心** (`redis_cache.cpp`) — 三级防护串联：

1. 布隆过滤器检查 → 不存在直接返回
2. 熔断器检查 → OPEN 则跳过 Redis 直连 DB
3. 缓存未命中 → SETNX 互斥锁 → 单线程重建

**熔断器** (`circuit_breaker.h`) — 三态状态机，`std::atomic` 无锁实现。

## 设计 FAQ

### Q1: 为什么选 Cache Aside 而非 Read/Write Through？

考研成绩是一次写入、永不修改的数据。Cache Aside 模式下成绩发布后缓存永不过期，写操作不存在（成绩不会在缓存有效期内被修改），不需要缓存-DB 同步机制。Read/Write Through 需要缓存层代理，增加复杂度而收益为零。

### Q2: 为什么用布隆过滤器而非 Redis Sorted Set 去重？

| | 布隆过滤器 | Sorted Set |
|:---|:---|:---|
| 100万考生内存 | ~1.2 MB | ~50+ MB |
| 查询延迟 | <1μs（本地内存） | ~0.1ms（网络往返） |
| 空间比 | **1 : 40** | — |

### Q3: 为什么 SETNX 而非 Redis 分布式锁框架？

Redisson/RLock 依赖 Lua + WatchDog，适合复杂分布式协调。本系统仅需缓存重建互斥：SETNX + TTL 足够（锁粒度是单个 key），无需引入额外依赖。获取失败时重试读缓存，而非阻塞等待。

### Q4: 为什么熔断器和布隆过滤器放在客户端？

- **网络延迟**：客户端内存操作 <1μs，Redis 网络往返 >0.1ms（差100倍）
- **故障隔离**：若 Redis 本身故障，集群层熔断器同样不可用
- **资源利用**：布隆过滤器 1.2MB 放客户端内存几乎零成本
